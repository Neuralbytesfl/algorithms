My experiment: faster sorting for 32-bit integers.

I was not trying to replace every sorting algorithm. I had a narrower question: if the data is only 32-bit integers, can I do better than a general-purpose comparison sort? With help from AI, I built and tested that idea.

178M integers per second on the best sustained short run for the stable single-thread radix16_reuse path.

The constraint changed the problem.

A normal sort must work for almost anything: objects, strings, custom comparison functions, and mixed data. My utility focuses on one shape: random 32-bit signed integers.

The optimization: stop comparing every pair.

For plain integers, we can sort by digits/buckets instead of asking “is A less than B?” over and over. That is why radix and counting strategies can beat general comparison sorts.

Comparison Sort

Works on any comparable object. Used by std::sort.

47 ms 1M random ints, measured here

Radix Sort

Integer-specific. Sorts using fixed-width key chunks.

5-6 ms best single-thread path

Adaptive Utility

Detects sorted, reverse, compact range, or random-like data.

8x sustained speedup over std::sort

What radix16 reuse does differently.

Convert signed order into sortable unsigned keys. Each value is transformed with value ^ 0x80000000, so negative and positive signed integers sort correctly by unsigned digit order.
Split each 32-bit integer into two 16-bit chunks. The low 16 bits form one bucket pass, then the high 16 bits form the second pass. That means only two data passes are needed for the key width.
Count buckets, prefix them, then scatter values. The algorithm counts how many values land in each bucket, converts counts into output positions, and writes values directly into their sorted regions.
Reuse scratch memory across sort calls. The fast utility keeps the temporary array and bucket counters ready instead of reallocating them every time. That reduces memory pressure during repeated workloads.

Why it works.

Less branching

std::sort repeatedly asks comparison questions. Radix sorting mostly performs predictable counting and memory writes.

Fixed number of passes

For 32-bit integers, the key width is fixed. The sorter does two 16-bit passes instead of growing with comparison complexity.

Tradeoff

It uses extra memory for scratch buffers and counters. The 60-second run peaked around 43 MB RSS.

Boundary

This is not a universal replacement. For objects, strings, custom comparators, or stable ordering rules, general sorts still matter.
